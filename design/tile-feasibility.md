# Tile Feasibility Test

Дата: 2026-05-08.

Цель: до реализации боевой сцены проверить, складывается ли идея tile-placement боя в принципе: можно ли придумать стартовую колоду, случайно раздать руки, замкнуть цветные границы, захватить землю, нанести урон и теоретически проходить уровни.

## Модель теста

Скрипт: `scripts/simulate-tiles.js`.

Параметры:

- поле 6x6;
- рука 7 тайлов;
- стартовая колода 36 тайлов;
- 100 сглаженных рук: лучшая из 3 кандидатных раздач;
- 100 симуляций каждого теоретического боя;
- 250 случайных попыток сборки на каждую кандидатную руку;
- урон захвата = площадь * 2;
- в бою невыбранные кандидатные руки уходят в сброс, выбранная рука играется как обычная;
- если цвет перебил атаку, во врага идет вся закрытая сумма цвета;
- если цвет не перебил атаку, игрок получает недостачу.

Временная модель тайла для теста: каждый тайл — сетка 3x3 микроземель. Края тайлов матчятся по 3-символьной подписи. Актуальная версия симуляции считает не закрытые цветные компоненты, а захват площади цветной границей.

Это не финальный формат тайлов, но хорошая проверка топологии: есть ли замыкаемые зоны, можно ли их растить, насколько часто рука мертвая.

## Обновление правила подсчета после арт-проверки

После первого пакета ассетов стало понятно, что center-exit тайлы визуально читаются как цветные границы, а не как готовая закрашенная земля. Для MVP принимаем более "земельное" правило:

```text
цветная граница замкнулась -> внутренняя область заливается этим цветом -> очки считаются по площади
```

Технически симуляция считает захват через flood fill:

1. Собрать все поставленные тайлы в общий `18x18` micro-grid для поля `6x6`.
2. Для каждого боевого цвета отдельно считать micro-cells этого цвета стеной.
3. Запустить flood fill по всем клеткам, которые не являются этим цветом, начиная с внешней рамки.
4. Все нецветные micro-cells, до которых flood fill не добрался, считаются захваченной внутренностью. Это включает и поставленные клетки, и пустую внутренность без тайлов.
5. Сила цвета считается по площади `граница + захваченная внутренность`.

Важные уточнения для симуляции:

- используется 4-соседство, диагонали не закрывают область;
- пустые macro-cells поля считаются внешним воздухом только если связаны с внешней областью;
- пустая область внутри полностью замкнутой границы становится захваченной землей и дает очки;
- поставленные клетки других боевых цветов внутри замкнутой границы считаются захваченной внутренностью для текущего цвета;
- красный, синий и зеленый считаются независимо;
- визуальная заливка внутренности не меняет edge signatures тайлов;
- старая статистика ниже полезна как проверка выкладываемости и базовой топологии, но баланс урона пересчитан под новое правило площади.

## Стартовая колода v2 для следующей проверки

На каждый боевой цвет — красный, синий, зеленый:

- `line_h` — горизонтальная линия;
- `line_v` — вертикальная линия;
- `corner_ur`, `corner_rd`, `corner_dl`, `corner_lu` — повороты.
- `tee_u`, `tee_r`, `tee_d`, `tee_l` — T-развилки;
- `plus` — перекресток границы, замена старого `dot`.

Плюс 3 серых пустых тайла.

Всего: 36 тайлов.

В v2 нет обычного `dot` с цветной клеткой только в центре: такой тайл не имеет выхода на край и не помогает строить границу. Базовые `cap`-тайлы тоже убираются из стартового набора, потому что они часто создают обрубки вместо замыканий. Их можно вернуть позже как special/low-power тайлы.

## Результаты сглаженной v2 capture-fill

Симуляция использует `assets/tiles_v2/tile_manifest.json`, строгий edge matching как основной режим, `damage = area * 2` и сглаживание раздачи через выбор лучшей из 3 кандидатных рук.

### Строгое совпадение краев

100 сглаженных рук:

- в среднем ставится 7.0 из 7 тайлов;
- все 7 тайлов поставлены в 97 из 100 рук;
- среднее число захватов: 0.3;
- средний урон: 6.0;
- 75 из 100 рук не нанесли урона;
- максимальный урон руки: 24.

Теоретические бои без апгрейдов:

```text
battle_01: 79/100 побед
battle_02: 39/100 побед
battle_03: 17/100 побед
battle_04:  2/100 побед
battle_05:  0/100 побед
```

Вывод: для первого боя это лучше подходит под 3-минутный MVP. Ненулевые руки встречаются чаще 6%, максимальный удар больше не ваншотит стартового врага на 35 HP, а поздние бои без апгрейдов не становятся автопроходимыми.

### Свободная adjacency-модель

100 сглаженных рук:

- в среднем ставится 7.0 из 7 тайлов;
- все 7 тайлов поставлены в 100 из 100 рук;
- среднее число захватов: 0.1;
- средний урон: 2.9;
- 88 из 100 рук не нанесли урона;
- максимальный урон руки: 24.

Теоретические бои:

```text
battle_01: 28/100 побед
battle_02:  2/100 побед
battle_03:  1/100 побед
battle_04:  0/100 побед
battle_05:  0/100 побед
```

Вывод: свободная adjacency-модель по-прежнему хуже строгой для этой эвристики. Для MVP остаемся на строгом edge matching.

## Что это меняет

1. Базовая формула для первого прототипа: `damage = captured area * 2`, а не `area * area`.

2. Первый бой должен использовать сглаживание стартовой руки/редроу: выбрать лучшую из 3 кандидатных раздач, остальные отправить в сброс. Это дает игроку больше ранних маленьких захватов, не ломая идею колоды.

3. Для первого боя текущие теоретические значения остаются приемлемыми: `35 HP`, атаки `1-2`, стартовый игрок `45 HP`.

4. Поздние бои в этой проверке не обязаны быть честными без апгрейдов. Их баланс нужно возвращать после реализации наград/прогрессии.

## Архив: v2 capture-fill до сглаживания

Симуляция использует `assets/tiles_v2/tile_manifest.json`: на каждый боевой цвет есть `line_h`, `line_v`, четыре `corner`, четыре `tee` и `plus`. Пустая область внутри полностью замкнутой цветной границы считается захваченной землей.

### Строгое совпадение краев

100 случайных рук:

- в среднем ставится 6.9 из 7 тайлов;
- все 7 тайлов поставлены в 88 из 100 рук;
- среднее число захватов: 0.1;
- средний урон: 8.6;
- 94 из 100 рук не нанесли урона;
- максимальный урон руки: 144.

Теоретические бои без апгрейдов:

```text
battle_01: 69/100 побед
battle_02: 61/100 побед
battle_03: 41/100 побед
battle_04: 34/100 побед
battle_05: 17/100 побед
```

Вывод: v2 ожил. Строгое совпадение краев не душит выкладку, и первый бой стал проходимым без апгрейдов. Но урон слишком бинарный: большинство одиночных рук ничего не закрывает, зато редкий захват дает 144 урона и часто решает бой одним взрывом.

### Свободная adjacency-модель

100 случайных рук:

- в среднем ставится 7.0 из 7 тайлов;
- все 7 тайлов поставлены в 100 из 100 рук;
- среднее число захватов: 0.1;
- средний урон: 7.2;
- 95 из 100 рук не нанесли урона;
- максимальный урон руки: 144.

Теоретические бои:

```text
battle_01: 30/100 побед
battle_02: 21/100 побед
battle_03: 11/100 побед
battle_04: 11/100 побед
battle_05:  6/100 побед
```

Вывод: свободная adjacency-модель больше не является автопобедой. В этом эвристическом прогоне она даже слабее строгой модели, потому что случайная выкладка без edge constraints чаще расползается и реже формирует полезные кольца. Для MVP остаемся на строгом edge matching.

## Что показала старая проверка

1. v2-набор `plus/tee/line/corner` работает лучше старого `dot/cap` набора: capture-fill теперь дает победы и большие payoff-моменты.

2. Главная проблема теперь не отсутствие захватов, а волатильность. 94-95% одиночных рук не наносят урон, но редкий захват бьет на 144.

3. Перед боевой сценой нужно сгладить первые минуты:

   - добавить больше маленьких guaranteed/near-guaranteed замыканий;
   - снизить или изменить формулу `площадь * площадь`;
   - ограничить стартовые enemy HP/атаки под rare big hit;
   - проверить стартовую руку/редроу/маленький обучающий capture.

4. `dot` не возвращать в MVP-колоду как обычный тайл. Если нужен маленький стабильный урон, лучше сделать отдельный special-тайл с явным правилом, а не скрытую центральную клетку.

## Архив: старая компонентная модель

Эти результаты были получены до перехода на правило "граница захватывает землю". Они полезны как проверка выкладываемости и волатильности старой топологии, но больше не являются балансом MVP.

### Строгое совпадение краев

100 случайных рук:

- в среднем ставится 7.0 из 7 тайлов;
- все 7 тайлов поставлены в 99 из 100 рук;
- среднее число закрытых зон: 0.9;
- средний урон: 9.4;
- 35 из 100 рук не нанесли урона;
- максимальный урон руки: 101.

Теоретические бои без апгрейдов:

```text
battle_01: 64/100 побед
battle_02: 40/100 побед
battle_03: 22/100 побед
battle_04:  6/100 побед
battle_05:  2/100 побед
```

Вывод: идея работает, но базовая колода очень волатильная. Это хорошо для риска, но первый бой надо делать мягче, а дальнейшие бои должны предполагать апгрейды. Без апгрейдов 5 боев не должны проходиться стабильно.

### Свободная adjacency-модель

100 случайных рук:

- все руки нанесли урон;
- среднее число закрытых зон: 3.5;
- средний урон: 54.3;
- нулевых рук нет.

Теоретические бои:

```text
battle_01: 100/100 побед
battle_02: 100/100 побед
battle_03: 100/100 побед
battle_04: 100/100 побед
battle_05: 100/100 побед
```

Вывод: если разрешить слишком свободное соседство, риск почти исчезает. Игрок регулярно получает много закрытых зон, и бой превращается в автопобеду.

## Архив: старое решение для MVP

1. Делать MVP со строгой совместимостью краев, но не с одноцветными квадратами.

   Нужны тайлы с внутренней 3x3 или похожей структурой: линии, концевики, углы, острова. Иначе закрытые зоны либо невозможны, либо слишком случайны.

2. Не начинать с атак вроде `красный 5, синий 3, зеленый 7`.

   Для стартовой колоды первые угрозы должны быть в диапазоне 1-3. Числа 5-8 — это уже зона после апгрейдов или поздние бои.

3. Оставить 7 тайлов в руке для первого прототипа.

   По симуляции рука почти всегда выкладывается полностью, значит edge-matching не душит саму выкладку.

4. Добавить безопасные маленькие закрытия.

   `dot`-тайлы дают игроку минимальный урон и понятный пример закрытой зоны. Без таких тайлов будет слишком много пустых раундов.

5. Принять волатильность как часть дизайна, но сгладить первые 3 минуты.

   35% нулевых рук для первого боя многовато. Возможные меры:

   - увеличить долю маленьких островов;
   - дать первый бой с низкими атаками;
   - дать игроку одну гарантированную закрытую зону в стартовой руке;
   - разрешить один бесплатный redraw руки;
   - позже добавить артефакты/апгрейды, которые снижают риск.

6. Считать свободную adjacency-модель запасным вариантом, а не основой.

   Она полезна как emergency fallback, если строгая топология окажется слишком сложной для UX, но в текущем виде она убивает напряжение.

## Что делать перед первой боевой задачей

Реализовывать боевую сцену надо не вокруг абстрактных цветных квадратов, а вокруг конкретного формата тайла:

```text
tile = 3x3 micro-land matrix
edge signature = 3 colors per side
capture boundary = micro-cells одного боевого цвета
captured interior = пустые или поставленные micro-cells, отрезанные boundary от внешнего воздуха
damage = (boundary cells + captured interior cells) * 2
```

Актуальная проверка показывает, что формат micro-grid и v2-набор тайлов годятся для MVP при строгом edge matching, линейном уроне `area * 2` и сглаживании руки через лучшую из 3 кандидатных раздач.
