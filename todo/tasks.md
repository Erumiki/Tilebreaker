### [2026-05-08] День 3: пересобрать стартовую колоду через anti-small-square recipe

**Idea:** заменить стартовую колоду "по одному тайлу из manifest" на явный recipe по `color + pattern`: меньше ранних замыкателей, больше линий/tee/серого fill.

**Why:** пока в колоде лежит полный набор corner/plus каждого цвета, рука часто предлагает готовый маленький квадрат. Большая зона должна рождаться из продолжения контура, а не из ожидания четырех углов.

**MVP:**

- добавить config-driven `startingDeckRecipe` или эквивалентный список count по tile id/pattern;
- MVP-рецепт без нового арта: уменьшить долю `corner`/`plus`, увеличить `line_h`, `line_v`, `tee_*`, добавить 4-5 `gray blank`;
- убедиться, что награды `add_tile`/`remove_tile` корректно работают с дублями tile id в колоде;
- симулятор должен читать тот же recipe, что и игра;
- не удалять тайлы из manifest и не менять арт.

**Acceptance:** симуляция показывает меньше quick 4-corner loops и меньшую долю `area <= 12`, средняя площадь закрытия растет хотя бы выше 13, `zero damage` не становится хуже baseline больше чем на 10-15%, первый бой остается проходимым в smoke/debug-прогоне.

**Priority:** must

**Layer:** MVP

### [2026-05-08] День 3: сделать opening shape-bag с cap на corner/plus

**Idea:** первые 8-12 выдач боя брать не чистым shuffle, а из shape-bag, где ограничены `corner` и `plus`, гарантированы разные цвета и есть продолжатели `line/tee`.

**Why:** даже после recipe чистый shuffle может выдать "либо ноль, либо готовую коробку". Opening bag должен дать игроку стартовый контур и выбор, а не готовое закрытие из руки.

**MVP:**

- добавить config `drawBag` с окном 8-12 выдач, `patternCaps` для `corner`/`plus`, минимумами для `line/tee` и отдельной дозировкой `gray`;
- применить bag только к старту боя/ранним выдачам, не ко всему забегу;
- сохранить seedable random для smoke/debug;
- симулятор должен печатать состав первых 8-12 выдач и сравнение `current recipe` vs `opening bag`;
- не возвращать hidden loop guarantee и не писать solver, который собирает игроку петлю.

**Acceptance:** в первых 8-12 выдачах нет готовой пачки замыкателей одного цвета, но есть валидные продолжения; доля ранних `area <= 12` падает, legal placement не проседает ниже baseline больше чем на 10%, `zero damage` не хуже baseline больше чем на 10-15%.

**Priority:** must

**Layer:** MVP

### [2026-05-08] День 3: попробовать queue-режим "текущий тайл + следующий" с теми же метриками

**Idea:** вместо выбора из полной руки дать игроку текущий тайл и preview следующего, чтобы решение было "куда встроить сейчас с учетом следующего", а не "какой набор сразу закрывает 2x2".

**Why:** полная рука превращает ход в поиск готовой коробки. Queue может заставить строить от доски и повысить ценность незакрытого контура.

**MVP:**

- добавить config-переключатель `drawMode`: `hand` или `queue`;
- в `queue` показывать текущий тайл крупнее и следующий тайл как preview;
- ставить можно только текущий тайл;
- после постановки preview становится текущим, добирается новый preview из draw pile/bag;
- конец раунда после фиксированного числа сыгранных тайлов или по кнопке;
- `discard` получает сыгранные и пропущенные queue-тайлы по понятному правилу;
- симулятор должен считать для queue те же метрики: `minimal capture share`, `zero damage`, `avg capture area`, `dead-end`.

**Acceptance:** игрок видит текущий тайл и следующий, ставит текущий, preview сдвигается вперед; сравнение показывает, стал ли queue реже приводить к `area <= 12` закрытиям без резкого роста нулевых раундов; smoke-тест проходит активный `drawMode`.

**Priority:** must

**Layer:** MVP

### [2026-05-08] День 3: сравнить hand recipe/bag vs queue и выбрать MVP-режим

**Idea:** быстро наиграть и измерить два кандидата: улучшенный `hand` с recipe/bag и `queue` с теми же источниками выдачи.

**Why:** нельзя полировать две разные игры. Нужен один основной режим, где ход чаще дает интересный выбор и реже сводится к минимальному квадрату.

**MVP:**

- пройти хотя бы по 2-3 коротких run в `hand` и `queue`;
- для каждого режима записать ощущения: есть ли выбор, понятен ли риск/награда, насколько часто игрок просто ждет угол;
- сравнить симуляторные метрики: `minimal capture share`, `avg capture area`, `zero damage`, `dead-end`, win rate, player damage;
- зафиксировать решение в `design/decisions.md` и `todo/current.md`;
- альтернативный режим оставить выключенным только если он не усложняет код и smoke.

**Acceptance:** выбран один активный `drawMode` для MVP; выбранный режим имеет ниже долю `area <= 12` закрытий или лучший ручной feel без неприемлемого роста нулевых раундов; решение записано в документации.

**Priority:** must

**Layer:** MVP

### [2026-05-08] День 3: если выдача не решит проблему — ослабить малый контур

**Idea:** сделать area-12 закрытие безопасным, но слабым ходом: оно помогает не получить полный ноль, но не является лучшей атакой.

**Why:** если recipe/bag/queue все равно оставляют 2x2 доминирующей стратегией, значит нужно менять не только выдачу, но и ценность маленького закрытия.

**MVP:**

- добавить config-driven `smallZonePenalty` для `area <= 12`: множитель, cap урона или перевод части value в shield/setup;
- показать в UI результата короткий лейбл "малый контур";
- 3x3/gray сценарий должен явно перебивать 2x2 по итоговому value;
- включать этот рычаг только после проверки recipe/bag/queue, чтобы не наказать игрока за единственный доступный ход.

**Acceptance:** 2x2 больше не лучший ответ по урону, но все еще лучше полного нуля; 3x3 с серым внутри дает минимум в 2 раза больше value, чем 2x2; smoke-тест проходит.

**Priority:** must

**Layer:** MVP
