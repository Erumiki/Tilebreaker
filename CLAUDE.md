# Tilebreaker

Игровой проект «Tilebreaker».

## Структура проекта

```
Tilebreaker/
├── src/
│   ├── core/       — ядро игры (игровой цикл, рендерер, ввод, менеджер сцен)
│   ├── entities/   — игровые сущности (игрок, блоки, бонусы, снаряды)
│   ├── render/     — системы рендеринга (атлас, спрайт-батчинг, эффекты)
│   ├── scenes/     — экраны и сцены (меню, игра, пауза, результаты)
│   └── utils/      — общие утилиты и хелперы
├── assets/         — игровые ресурсы (спрайты, звуки, шрифты, тайлмапы)
├── configs/        — конфигурационные файлы (баланс, уровни, UI, правила)
├── design/         — геймдизайн-документация (GDD, UI, механики, прогрессия)
├── techspec/       — технические спецификации (рендеринг, формат атласа)
└── todo/           — задачи и баги для программиста
```

## Технический стек

- **HTML + Vanilla JS + Pixi.js** — без UI-фреймворков и без полноценных игровых движков
- **Vite** — основной dev server и резолвинг npm-зависимостей
- **Playwright** — smoke-тесты playable build в браузере
- **Весь игровой рендеринг через canvas/Pixi.js**, включая UI — не использовать DOM-элементы для игрового интерфейса
- Единая точка входа — `index.html`, вся логика в `src/`
- Код и документация на русском, комментарии в коде на английском

## Локальные инструменты

Node.js LTS установлен локально в `.tools/` и не коммитится. Для npm/node использовать wrapper-скрипты:

```sh
./scripts/npm.sh run dev
./scripts/npm.sh run check
./scripts/npm.sh run test:e2e
```

Если команда напрямую требует `node` или `npm`, сначала добавить локальный Node в `PATH` через wrapper или явный `env PATH=...`.

После изменений, которые затрагивают playable build, сцены, UI или frontend-поведение, Codex сам запускает локальный dev server через `./scripts/npm.sh run dev`, открывает актуальный локальный URL в браузере для пользователя и сообщает адрес. Не перекладывать этот шаг на пользователя, если запуск не заблокирован sandbox/approval или сломанной сборкой.

## Пайплайн рендеринга

- **Pixi Application** — единая точка рендеринга, resize и ticker.
- **Pixi Containers** — сцены, UI-слои, игровые слои и будущие эффекты собираются через контейнеры.
- **Pixi Graphics/Text/Sprites** — кнопки, панели, текст, игровые HUD-элементы и будущие спрайты рисуются внутри canvas.

## Рабочий процесс команды

- `main` — стабильная ветка, мержить только через Pull Request
- `feature/<name>` — новые фичи
- `fix/<name>` — исправления багов
- PR должны быть маленькими и сфокусированными

## Game Jam Workflow

Во время геймджема все новые идеи и решения проходят через три роли:

- **ГД** — расширяет идею, проверяет её на целостность, формулирует фантазию, эмоцию игрока и место идеи в общем дизайне.
- **Разработчик** — проверяет реализуемость, режет скоуп, предлагает минимальную версию и критерий готовности.
- **Лид** — принимает финальное решение, следит за playable build, приоритетами и правилами проекта.

При начале любой новой фичи, балансовой итерации или крупной UI-правки Codex работает как **лид** и поднимает минимум двух фоновых агентов:

- **ГД-агент** — следит, чтобы решение реально игралось: есть понятная фантазия хода, риск/награда, читаемый результат действия, связь с core-loop и первый 3-минутный опыт не превращается в сухую проверку правил.
- **Разработчик-упроститель** — следит, чтобы реализация не становилась слишком сложной: режет лишние системы, ловит преждевременные абстракции, предлагает самый короткий путь к acceptance и защищает цикл `меню -> битва -> результат -> улучшение -> следующая битва -> финал`.
- **Лид Codex** — держит задачу, принимает или отклоняет советы агентов, интегрирует итоговое решение, проверяет билд и отвечает пользователю за конечный результат.

Агентский цикл обязателен для любой нетривиальной работы: новой фичи, балансовой итерации, крупной UI-правки, изменения core-loop, UX, экономики или правил выдачи/скоринга. Пользователь не должен вручную напоминать Codex привлекать агентов.

Минимальный процесс:

1. Codex формулирует текущую задачу и просит ГД-агента и разработчика-упростителя дать позицию.
2. Агенты коротко сверяют позиции между собой: что спорно, что оставить в MVP, что отложить.
3. Codex как лид сверяет их выводы со своей позицией, принимает финальное решение и объясняет, что берет в работу.
4. Только после этого Codex переходит к правкам, реализации или финальным рекомендациям.

Исключения из полного агентского цикла: tiny-команды, исправления опечаток/текста, механические рефакторы без влияния на геймплей/UI/дизайн, простая диагностика, одноразовые ответы или явная просьба пользователя пропустить агентское обсуждение. Если Codex пропускает агентский цикл, причина должна быть очевидной: задача мала, локальна и не меняет игровые решения.

Фоновые агенты по умолчанию не меняют файлы. Они дают короткий фидбек лидеру: `играется/не играется`, `сложно/нормально`, главный риск, минимальная правка. Если для задачи нужен дополнительный агент, сначала дописать его роль и ответственность в этот раздел, затем использовать в работе.

Минимальный формат отчета ГД-агента:

```text
ГД-проверка:
- Играбельность: проходит / не проходит.
- Усиливает core-loop: да / нет.
- Риск для playable build: ...
- Что оставить в MVP: ...
- Что отложить: ...
- Вердикт: можно брать / надо резать / стоп.
```

Минимальный формат отчета разработчика-упростителя:

```text
Упростил: ...
MVP: ...
Риск/стоп-сигнал: ...
```

Если пользователь явно говорит как **дизайнер**, не менять код. Разрешены только дизайн-документы, арт-задачи, баланс-идеи, постановки задач и другие `.md`-материалы.

Если пользователь явно говорит как **разработчик**, менять только код и технические файлы. Не расширять дизайн-решения без отдельного запроса.

Если роль не указана, считать пользователя лидом/продюсером: обсуждать решение, фиксировать правила или задачи, но не менять код без явного технического запроса.

Дизайнер общается с разработчиком только через постановки в `.md`: задачи, acceptance criteria, приоритеты и ссылки на дизайн-контекст.

После завершения работы фоновых агентов обязательно закрывать их треды. Это сохраняет лимит агентов свободным и предотвращает блокировку следующих обсуждений.

Если в работе участвовали фоновые агенты, перед финальным ответом собрать их результаты, сверить расхождения, закрыть треды и дать пользователю финальный фидбек: что каждый агент сделал, что принято в итоговое решение, что отклонено или отложено и почему.

## Scope Control

- Любая новая идея должна иметь `MVP`: минимальную проверяемую версию на 30-90 минут работы.
- Хорошие идеи не удаляются: они получают слой `MVP`, `Jam Stretch` или `Post-jam`.
- В работу попадает только `MVP`, если он усиливает текущий playable build.
- Все задачи маркируются приоритетом `must`, `nice` или `dream`.
- Если задача не улучшает первый 3-минутный опыт игрока, она должна отдельно доказать ценность.
- Каждый рабочий день должен заканчиваться кликабельным билдом.
- Нельзя оставлять основной цикл в сломанном состоянии.
- Пока не работает цикл `меню -> битва -> результат -> улучшение -> следующая битва -> финал`, не добавлять новые режимы, глубокую мету и крупные параллельные системы.
- Единственный список задач, порядок работы, следующий рабочий пункт и статус выполнения ведутся в `todo/tasks.md`.
- Следующую задачу выбирать только из `todo/tasks.md`: первый незачеркнутый заголовок сверху вниз.
- `todo/current.md` — только снимок текущей версии и дизайн-контекст. Не хранить в нем task list, build order, next task, acceptance или статусы задач.
- Если где-либо кроме `todo/tasks.md` встречается next-step или список задач, считать это устаревшим контекстом и перенести/удалить дубликат.
- Новые идеи, которые появились во время работы, но не входят в текущий план, записываются в `todo/thoughts.md`. Периодически при планировании переносить подтвержденные мысли в `todo/tasks.md` или удалять/закрывать как неактуальные.
- После выполнения задачи пройтись по релевантной документации (`README.md`, `todo/current.md`, `design/`, `techspec/`, `CLAUDE.md`) и обновить устаревшие сведения о том, где что хранится, как запускается, какие правила действуют и какие решения теперь являются текущей правдой.
- Закрытые версии и snapshots фиксируются в `history/`.

## Формат задач

Новые задачи в `todo/tasks.md` должны быть короткими, но actionable:

- `Idea` — что хотим сделать.
- `Why` — зачем это игроку.
- `MVP` — минимальная версия.
- `Acceptance` — критерий готовности в формате: игрок видит X, нажимает Y, получает Z.
- `Priority` — `must`, `nice` или `dream`.
- `Layer` — `MVP`, `Jam Stretch` или `Post-jam`.

## Конвенция коммитов

Используем Conventional Commits:

- `feat:` — новая функциональность
- `fix:` — исправление бага
- `docs:` — изменения документации
- `refactor:` — рефакторинг без изменения поведения
- `chore:` — сборка, конфиги, зависимости
- `style:` — форматирование без изменения логики

## Конфигурация и баланс

Все значения баланса, тюнинга и настроек хранятся в `.json` файлах в `configs/`. Никогда не хардкодить значения баланса в исходном коде: скорость шарика, здоровье блоков, урон, цены, дроп-рейты, тайминги, размеры и UI-позиции должны читаться из конфигов.

При изменении **структуры** JSON-конфигов (добавление/удаление/переименование полей — не значений) обязательно сообщать об этом КАПСЛОКОМ в ответе. Например: «ДОБАВЛЕНО ПОЛЕ `ballSpeed` В `game.json`» или «УДАЛЕНО ПОЛЕ `dropRate` ИЗ `bonuses.json`».

## Баги для программиста

Если обнаруживается баг в коде игры, который нужно исправить программисту, записывай его в `todo/bugs.md`. Формат: дата, краткое описание, что именно нужно поменять и в каком файле.

Не исправляй код самостоятельно без явного разрешения, если задача была только на документацию, анализ или постановку ТЗ.

## Задачи для программиста

Если нужно поставить задачу на новую фичу или улучшение (не баг), записывай её в `todo/tasks.md`. Формат: дата, название, описание что нужно сделать, где смотреть в коде, зачем это нужно.

**Важно:** баги и задачи — разные файлы. Баг — это что-то сломанное. Задача — это что-то новое или запланированное.

## Закрытие багов и задач

При исправлении бага или выполнении задачи — стереть тело описания и пометить заголовок как выполненный: обернуть в `~~зачёркивание~~` и дописать статус.

Примеры:

```
### ~~[2026-05-08] Шарик застревает в углу~~ ИСПРАВЛЕНО
### ~~[2026-05-08] Экран результатов~~ ВЫПОЛНЕНО
```

Закрытые задачи не должны занимать экран.

## Чтение задач и багов

Перед началом любой работы по коду — читай эти файлы в таком порядке:

- `todo/tasks.md` — единственный backlog, порядок задач, следующий рабочий пункт и статусы.
- `todo/current.md` — текущая версия и дизайн-контекст без списка задач и acceptance.
- `todo/bugs.md` — известные баги.
- `todo/thoughts.md` — несинхронизированные идеи и наблюдения, которые еще не стали задачами.

Там может быть контекст, который влияет на то, что и как делать.

Если во время работы появляется хорошая идея вроде изменения JSON-конфигов, сначала сравни ее с `todo/tasks.md`. Если задача уже есть в плане — обнови соответствующую задачу в `todo/tasks.md`. Если задачи нет — запиши мысль в `todo/thoughts.md`, а не теряй ее в финальном сообщении.

## Архитектурные решения

Все значимые решения логируются в `design/decisions.md` с датой, контекстом и обоснованием. Проверяй этот файл перед архитектурными изменениями.

## Технические спецификации

Подробная техническая документация хранится в `techspec/`. Сверяйся с ней перед изменением рендеринга, атласа, батч-системы, форматов уровней и загрузки ассетов.
